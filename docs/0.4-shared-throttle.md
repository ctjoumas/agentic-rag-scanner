# Task 0.4 - Shared Throttle (`ISharedThrottle`)

> Phase 0 / Epic 0, story 0.4. Quick background for PR review.
> See also: `horizon-scanner-architecture.md` (Open considerations -> "Shared quota contention"),
> `implementation-plan.md` Phase 0, and `backlog.md` Epic 0 / 0.4.

## Why this exists
The scanner runs one agentic-RAG workflow **per topic group**. They run sequentially today but will
run **in parallel** later (Phase 12). Every workflow hits the **same** shared services - Azure OpenAI
(TPM / RPM limits) and Bing (QPS limits) - so without a single shared gate, N concurrent workflows
would blow those quotas and trigger 429s.

`ISharedThrottle` is that single, injectable gate. All outbound, quota-limited calls funnel through
it, so we can cap how many / how often we hit OpenAI and Bing regardless of how many workflows are in
flight.

## Scope of THIS PR (Phase 0)
Only the **abstraction + a pass-through implementation**. There are **no real limits yet**, on
purpose:
- Phases 1-11 run sequentially, so contention isn't a problem yet.
- Wiring the interface now lets every call site be "throttle-ready" before we switch on real limits,
  with no later call-site churn.

The real limiter (built on `SemaphoreSlim` and/or `System.Threading.RateLimiting`, with per-service
TPM/RPM/QPS config) arrives in the hardening / parallelization phases. **Core stays package-free** -
the real implementation can wrap a `RateLimiter` without Core taking that dependency.

## The shape
```csharp
public interface ISharedThrottle
{
	// Waits until `permits` units of quota are free, then returns a lease. Dispose to release.
	ValueTask<IThrottleLease> AcquireAsync(int permits = 1, CancellationToken cancellationToken = default);
}

public interface IThrottleLease : IDisposable { }
```
- `permits` = 1 per request for RPM/QPS; pass an estimated token count for token-based limits (TPM).
- `NoOpThrottle` is the Phase 0 impl: `AcquireAsync` returns immediately and the lease's `Dispose`
  does nothing.

## The lease is a scope guard, not a parameter
The one non-obvious bit: the lease is **not passed into the LLM/Bing call**. It's a lifetime guard
(same idea as a lock). Throttling happens at the two ends - *acquire* (wait for room) and *dispose*
(release the slot); the call in between just runs.

```csharp
var lease = await throttle.AcquireAsync(ct);   // WAIT here if we're over the limit
try
{
	var resp = await chatClient.GetResponseAsync(...);   // the call doesn't know about the throttle
}
finally
{
	lease.Dispose();   // release the slot so the next waiter can proceed
}
```
Same pattern as `SemaphoreSlim.WaitAsync()` / `Release()`. Holding the lease = "I'm occupying a slot
of quota"; disposing it = "done, release it."

Dispose semantics: for **concurrency** limits it returns the slot (essential); for **time-based**
limits (RPM/QPS/TPM) permits refill on a timer, so dispose is a no-op there - but callers always
`using` it so the same code works for both.

## How to use it (everyday)
Prefer the `ExecuteAsync` extension - it does acquire/run/release for you, so call sites never touch a
lease:
```csharp
var resp = await _throttle.ExecuteAsync(
	ct => _chatClient.GetResponseAsync(prompt, ct),
	cancellationToken: token);
```
Drop down to `AcquireAsync` only if you need to hold a slot across something `ExecuteAsync` can't
express (e.g. a streamed response).

## What's in the PR
- `AgenticRagScannerApi.Core/Throttling/`
  - `ISharedThrottle.cs` - the gate (the `AcquireAsync` primitive)
  - `IThrottleLease.cs` - the disposable lease
  - `NoOpThrottle.cs` - Phase 0 pass-through impl

- DI: `services.AddSingleton<ISharedThrottle, NoOpThrottle>();` (API `ServiceCollectionExtensions`)
- Tests: `NoOpThrottleTests` - lease is disposable; `ExecuteAsync` runs the op and returns its result

## Naming note
The backlog originally called this `ILlmThrottle`; we renamed it to **`ISharedThrottle`** because it
fronts both OpenAI **and** Bing, not just the LLM.
